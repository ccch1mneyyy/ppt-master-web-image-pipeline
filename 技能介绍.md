# PPT Master Web Image Pipeline

## 这是什么

`ppt-master-web-image-pipeline` 是 `PPT Master` 的配套技能，专门解决一个很常见的痛点：

- 你已经用 `PPT Master` 做到生图阶段了
- `design_spec.md` 里也列好了需要哪些图
- 你有 Gemini、Nano Banana、ChatGPT、Grok、豆包这些网站的会员
- 但你没有 API key，也不想为了一张图单独充 API 额度

这个技能的作用就是：在 `PPT Master` 的 Step 5（图片生成）阶段，把工作流对接到你**已经登录的网页生图工具**上，让你不用 API 也能跑完整流程。具体来说它会做这些事：

1. 从 `design_spec.md` 里读出待生成的图片清单
2. 根据项目风格统一生成图片提示词
3. 指导（或配合浏览器自动化完成）网页端的图片生成
4. 处理下载、截图回退、水印和角标清理
5. 把生成结果按 `PPT Master` 要求放进 `project/images/`
6. 回写状态，让后续流程可以继续往下走

## 它适合谁

- 已经在用 `PPT Master` 的人
- 有网页端 AI 生图会员，但没有 API key
- 希望 PPT 里的图片风格统一、贴合页面，而不是随便出几张图就塞进去
- 想把"生成 → 命名 → 归档 → 回写状态"这套流程规范化

## 核心能力

- 支持 Gemini / Nano Banana / ChatGPT / Grok / 豆包
- 自动从 `design_spec.md` 的 `Image Resource List` 提取待生成项
- 自动生成提示词文档 `images/image_prompts.md`
- 自动生成任务清单 `images/web_generation_manifest.json`
- 根据图片类型推荐最适合的生图平台
- 约束文件命名，确保符合 `PPT Master` 的要求
- 处理网页端常见的下载失败、`blob:` 链接、截图裁切等问题
- 处理 Gemini / Banana 系的角标、水印、边缘裁切
- 最后校验图片是否存在、比例是否匹配、状态是否需要更新

## 工作流

### 1. 读取项目信息

技能会读取项目中的：

- `design_spec.md`
- Section I 的项目信息
- Section III 的视觉主题和配色
- Section VIII 的 `Image Resource List`

所有生成结果都会放到 `project/images/<filename>` 下，不会凭空造文件名，也不会随意改目录结构。

### 2. 生成提示词和任务清单

这一步会产出两个文件：

- `images/image_prompts.md` — 提示词，方便你人工检查和调整
- `images/web_generation_manifest.json` — 任务清单，给后续自动化流程用

### 3. 选择生图平台

根据图片类型给出推荐：

| 图片类型 | 推荐平台 |
|----------|----------|
| Background（背景） | Gemini / Nano Banana / Grok |
| Photography（摄影） | ChatGPT / Grok / 豆包 |
| Illustration（插画） | ChatGPT / 豆包 / Gemini / Nano Banana |
| Decorative（装饰） | Gemini / Nano Banana / ChatGPT |
| Diagram（示意图） | 只适合做概念草图，精确带标签的图表建议另寻他法 |

### 4. 网页生成与抓图

这一步需要配合 Codex 的浏览器能力，复用你已经登录的网页会话来完成。

抓图的优先级是这样的：

1. 首选：直接下载原图或全尺寸图片
2. 如果下载失败：尝试打开独立的图片预览视图再抓
3. 还不行：走 DevTools 抓 `blob:` 或网络请求中的图片
4. 最后兜底：截图 + 裁切

### 5. 清理水印和角标

带明显平台水印、logo、聊天界面的图片不会直接放进 PPT。

处理方式：

- Gemini / Nano Banana 系角标：配合 `PPT Master` 自带的 `gemini_watermark_remover.py` 处理
- 通用边缘角标、截图边框：用技能自带的 `postprocess_web_image.py` 裁切
- 如果水印压在图片主体上，优先重新生成，而不是强行修图

### 6. 校验并回写状态

最后做一轮检查：

- 图片文件是否确实存在
- 文件名是否正确
- 宽高比是否大致符合 `design_spec.md` 的要求
- 状态是否需要从 `Pending` 更新为 `Generated`

确认没问题后，把项目交还给 `PPT Master` 的 Executor 继续后续流程。

## 当前版本的定位

这个版本是一个**为浏览器自动化设计的工作流技能**：

- 项目侧的稳定逻辑已经封装成脚本
- 网页操作部分依赖 Codex 的浏览器能力
- 它不是一键全自动的网页机器人——但现在这个版本已经可以在 Codex 里实际使用了

如果后续想往"全自动浏览器执行器"方向升级（比如接入 Chrome DevTools MCP 或 Playwright），可以在此基础上继续扩展。

## 包内主要文件

```
ppt-master-web-image-pipeline/
├── SKILL.md                              # 技能主入口
├── references/
│   ├── ppt-master-bridge.md              # 与 PPT Master 的对接说明
│   ├── web-providers.md                  # 各生图平台的使用说明
│   └── capture-and-watermarks.md         # 抓图与水印处理参考
└── scripts/
    ├── extract_image_resource_list.py    # 从 design_spec 提取图片清单
    ├── build_ppt_image_prompts.py        # 生成提示词
    ├── update_design_spec_status.py      # 回写生成状态
    ├── verify_ppt_images.py              # 校验图片
    └── postprocess_web_image.py          # 后处理（裁切、去角标）
```

## 安装方式

把 `ppt-master-web-image-pipeline` 文件夹放到 Codex 的 skills 目录下即可：

```text
~/.codex/skills/
```

Windows 用户一般在这里：

```text
C:/Users/你的用户名/.codex/skills/
```

放好之后重启 Codex 让它重新加载技能。

## 推荐搭配

最佳组合：

- **PPT Master** — 负责 PPT 整体设计流程
- **ppt-master-web-image-pipeline** — 负责网页端图片生成
- **你已登录的网页生图账号** — Gemini / Nano Banana / ChatGPT / Grok / 豆包

这样即使没有 API key，也能把 `PPT Master` 的图片生成环节跑通。

这个分享包是一个面向真实工作流的技能包，不是演示空壳。里面已经包含可执行脚本、参考文档和完整的技能结构。

如果后续想增强，最有价值的方向是补上"浏览器自动执行器"层——比如基于 Chrome DevTools MCP 或 Playwright 的自动页面操作。
